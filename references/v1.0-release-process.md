# v1.0 Release Process — skill-sublation

This reference documents the concrete steps used to take a sublation-governed skill from "internal governance" to "public GitHub release." Validated during the skill-sublation v1.0 release (2026-05-29).

## Pre-Release Checklist

### 1. Governance Health Check

```bash
python3 scripts/lifecycle.py health --warn-after-days 7
python3 scripts/lifecycle.py scan
```

Requirements: queue empty, health clean, all relevant candidates closed.

### 2. Identify Governance Gaps

Scan for "unledgered formal drift" — content in the formal skill with no candidate manifest:

```bash
for d in ~/.hermes/sublation/candidates/<skill>/*/; do
  grep -l "<topic-keyword>" "$d/PATCH.diff" 2>/dev/null
done
# If empty, the content was added outside the pipeline — needs backfill
```

### 3. Backfill Any Gaps

Create a backfill candidate (see Pitfalls in SKILL.md for backfill policy):

```
candidate_id: <date>-<topic>-backfill-<agent>
retroactive_backfill: true
promotion_mode: user_delegated_agent_patch
status: closed (immediately — content already validated)
closure_reason: "backfill — content already validated in production"
```

Backfill audit will get `conditional` with `strict_formal_post_promotion_current` WARNING. This is expected — later candidates legitimately modified other files.

### 4. Sanitize for Public Release

```bash
# Remove bytecode pollution
find . -name "__pycache__" -type d -exec rm -rf {} + 2>/dev/null
# Remove backup files
find . -name "*.orig" -o -name "*.bak" | xargs rm -f
# Replace hard personal paths
grep -rn "/Users/" . --include="*.md" --include="*.json" --include="*.py"
# Replace all with ~/ equivalents
```

### 5. Prepare Repo Structure

```
<skill-name>-release/
├── README.md | CHANGELOG.md | LICENSE | .gitignore
├── RELEASE-v1.0.md | SKILL.md
├── scripts/ | schemas/ | references/
```

Exclude: `candidates/`, `rollback-points/`, `observations/` (runtime data, not source).

### 6. Codex + Hermes Joint Audit

See `references/pre-release-audit-pattern.md` for the full two-phase audit workflow.

### 7. GitHub Push

`gh auth login` in the Hermes terminal session may fail because the token is stored in the user's local keychain which the Hermes session can't access. **Workaround**: write a `publish.sh` script for the user to run in their own terminal:

```bash
#!/bin/bash
set -e
gh repo create <user>/<repo> --public --source=. --remote=origin --push
```

### 8. Post-Release

- Update memory with release date and repo URL
- skill-sublation enters maintenance mode
