# v1.x Pitfalls Archive

v2.0 将旧 `SKILL.md` 的长 Pitfalls 列表归档到这里。主文档只保留宪法、硬边界、流程和设计模式；细节经验按主题查阅。

## 阶段和节奏

- 筛选完不要急着建候选。先让用户决定候选顺序、scope 和期望产物。
- 验证闭环比候选数量重要。一个生产 smoke 通过的候选，比十个只 audit passed 的候选更有价值。
- 别急着跳阶段。先闭合当前阶段，再进入下一阶段。
- 不要过度治理 trivial update。元数据小改可以批处理，但仍要留账。

## 上游和漂移

- 先同步上游再筛选。对旧版本做 sublation 容易自欺。
- `hermes skills update` 返回 No updates 不代表 local source 最新。
- 上游同步后，已晋升候选的 `formal_post_promotion` 可能 drift；优先 backfill/repair，不要盲目 rollback。
- 顺序晋升会让同一 skill 的观察窗候选快照漂移；需要解释、刷新、闭合或建治理候选。
- 未入账的正式内容变更必须补 backfill。

## Manifest 和审计

- `source_skill.files` 必须包含正式技能全部文件，不只是改动文件。
- 晋升后必须补 `validation.formal_post_promotion`。
- `promotion_mode`、`empirical_scorecard.status`、`relationships.sublation_pattern` 必须使用 schema 枚举值。
- PATCH.diff 不能有 bare `@@` hunk header。
- framework audit passed 不等于业务 skill 能跑；还需要 business smoke。

## Agent 协作

- Hermes 和 Codex 有分歧时，先在群聊澄清，不要各自执行。
- Codex 超时不代表 Hermes 可越权写核心代码；若只是机械文档补齐可接手，核心实现仍交回 Codex。
- Codex 余额耗尽时，Hermes 可接手纯 spec-patch，但必须在 manifest 说明。
- Codex 需要可信 git/workspace 上下文；复杂 prompt 不要内联进 shell 单引号。

## 权限和外部系统

- 登录墙采用 human-in-the-loop，不保存、不读取、不转发凭据。
- 登录墙 MCP 预留是设计决策，不是长期 PENDING 缺陷。
- 外部 skill 平台可能有隐藏写入能力；先做源码级写入能力分析。
- GitHub 发布前必须清理 pycache、`.orig`、个人路径、内部样本，并完成联合审计。
- gh auth 跨 session 失败时，不要反复登录；可生成 `publish.sh` 让用户本地执行。

## 跨技能关系

- 吸收不等于低位阶吞并。先读双方能力，明确 target 大体系吸收 donor 的局部经验。
- `close-superseded` 会覆盖旧 notes，执行前先检查上下文。
- 已被吸收的 donor skill 后续 sublation 应路由到 umbrella。
- 外部源创建新技能候选时，source 可以是外部 clone，但目标安装路径必须在晋升前验证。

## 群聊桥

- 桥是协调通道，不是治理权力。
- POST `/api/messages` 必须用 `speaker` 字段；不要用 `from` 或 `source`。
- 本地桥现在以 LaunchAgent 常驻 `127.0.0.1:8787` 为主；历史的 `192.168.x.x` 拓扑问题只作为旧案参考。
- 不要刷屏；发现应汇总成一条有证据的消息。

## 非 sublation 任务

用户没提 sublation 时，不要主动把普通任务扩展为 sublation session。可以提示“需要时我可以看”，但不要自动开始候选审计或晋升推进。
