# skill-sublation v1.0 发布报告

发布日期：2026-05-29

## 1. 发布声明

`skill-sublation` 正式声明进入 `v1.0`。

这一版本标志着 skill-sublation 从实验性治理流程进入稳定维护阶段。框架已经具备完整的技能治理闭环：从真实运行中产生观测，形成候选，在候选层完成修改、审计和复核，经授权晋升到正式技能，并在观察窗口内确认没有回归后闭合。

自 `v1.0` 起，本项目进入维护模式。后续默认只接受缺陷修复、文档澄清、兼容性调整和治理账本补录；任何会改变生命周期语义、晋升权限、审计门槛或 schema 契约的改动，都必须进入新的版本线。

## 5. 已知限制与范围边界

skill-sublation 是技能治理框架，不是自动修复一切技能问题的系统。它能让变更可追踪、可审计、可回滚，但不能替代真实业务验证。

明确边界如下：

- 审计通过只表示候选结构和治理记录符合要求，不等于业务工作流一定成功。
- 候选、rollback point 和运行态日志属于治理运行数据，公开仓库只保留框架、schema、脚本和整理后的发布说明。
- Darwin 或其他外部评估器只能提供参考证据，不能绕过候选层、审计、复核和用户审批。
- 跨技能吸收默认吸收能力、接口和流程，不默认复制受保护表达或源文件。
- 原技能删除必须经用户明确审批；合并候选本身不能自动删除源技能。
- 外部服务、账号、MCP、网络、浏览器、cron 和生成后端相关能力必须通过真实或可解释的烟雾测试确认。
- backfill 只用于补录已经验证生效的治理链路，不能用来伪造尚未完成的复核或晋升。

## 6. 维护模式政策

### v1.0.x 允许的维护改动

- 修复脚本 bug，且不改变生命周期语义。
- 澄清文档、示例、错误信息和操作说明。
- 更新兼容性处理，例如路径差异、Python 版本兼容、只读检测和更稳健的报错。
- 补充历史案例、发布说明、CHANGELOG 或 backfill 记录。
- 改进非阻断型 warning 文案，前提是不改变候选是否可晋升的判断结果。
- 增加测试 fixture 或 smoke test 示例，前提是不引入新的强制治理规则。

### 需要新版本的改动

- 新增或删除生命周期状态。
- 修改 `observe → candidate → audit → review → promote → observation window` 主链路。
- 修改晋升权限、用户审批边界或源技能删除规则。
- 修改 manifest / observation schema 的必填字段或含义。
- 新增会阻断晋升的 strict audit gate。
- 放宽 rollback、目标路径验证、业务烟雾测试或旧能力保留要求。
- 允许外部评估器直接改正式技能、直接晋升或直接关闭候选。
- 改变跨技能吸收的权利来源要求。
- 将维护性 bug fix 扩展为新的治理能力。

版本策略：

- `v1.0.x`：维护修复，不改变治理协议。
- `v1.1`：向后兼容的新能力。
- `v2.0`：生命周期、schema、审计门槛或权限模型发生破坏性变化。

## 结论

skill-sublation v1.0 的核心承诺是：技能可以进化，但每一次进化都必须留下可检查的治理链路。

本版本完成了从单点修补到系统治理的闭环，并用真实候选、合并整理和补账机制验证了框架自身。自本发布起，skill-sublation 进入维护模式。
